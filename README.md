# Ad Platform — Go + Python

Это Ad Platform — проект, в котором я реализовал backend-архитектуру с нуля. Я сделал систему, которая управляет рекламными кампаниями, показывает аналитику и при этом является надёжной и масштабируемой. Для этого я разбил всё на отдельные сервисы, каждый из которых отвечает за свою задачу.

**Кратко о сервисах:**
- `auth-api` (Python/FastAPI) — пользователи, аутентификация, балансы, финансовые транзакции
- `ads-api` (Go/Gin) — управление кампаниями, объявлениями и креативами
- `analytics-api` (Python/FastAPI) — хранение и анализ статистики в ClickHouse
- `worker-go` и `worker-py` — асинхронная обработка событий из RabbitMQ

**Что я также реализовал в проекте:**
- Разделил OLTP (транзакции, пользователи, балансы) и OLAP (аналитика, статистика)
- Настроил асинхронную обработку событий через RabbitMQ
- Реализовал финансовые операции с гарантиями ACID (в auth-api)
- Добавил инфраструктуру для кэширования в Tarantool
- Настроил мониторинг через Prometheus и Grafana
- Настроил развёртывание через Ansible
---

## 🧱 Архитектура проекта

Я организовал проект как моно-репозиторий, внутри которого находятся несколько микросервисов. Это удобно: весь код в одном месте, но при этом сервисы независимы и могут разрабатываться отдельно.

---

## 🔹 Роли сервисов

### `auth-api` (FastAPI, Python)

Это основной сервис для работы с пользователями и транзакциями. Я сделал его на Python с FastAPI, потому что тут много бизнес-логики и важно быстро итерироваться.

**Что я здесь реализовал:**
- Регистрация и логин пользователей
- JWT-аутентификация с токенами
- Система ролей (кто админ, кто обычный пользователь)
- Управление балансами и финансовыми транзакциями
- Миграции базы данных через Alembic

**Самое важное:** все финансовые операции я обернул в ACID-транзакции. Это значит, что деньги не могут «потеряться» — если что-то идёт не так, вся операция откатывается назад.

**База данных:** я спроектировал схему с таблицами `users`, `balances`, `transactions`, `campaigns`, `ads`, `creatives`. Все таблицы связаны через внешние ключи (FK), чтобы данные были целостными.


### `ads-api` (Go)

Этот сервис я написал на Go для работы с рекламными кампаниями и объявлениями. Go выбрал потому, что тут важна скорость и эффективность при обработке множества запросов.

**Что я здесь сделал:**
- CRUD-операции для кампаний, объявлений и креативов (создание, чтение, обновление, удаление)
- Подключение к MySQL через библиотеку `sqlx`
- Разделил код на слои: handler → service → store (так проще тестировать и поддерживать)
- Отправку событий в RabbitMQ (когда объявление создаётся или обновляется)

**Важный момент:** этот сервис не владеет схемой базы данных и не делает миграции — этим занимается `auth-api`.

### `analytics-api` (FastAPI, Python)

Сервис для аналитики. Я сделал его отдельно, потому что аналитические запросы сильно отличаются от обычных транзакций — они читают много данных и агрегируют их.

**Что я здесь реализовал:**
- Хранение статистики по показах, кликах и расходах в ClickHouse
- REST API для получения аналитики с фильтрацией по датам, кампаниям, объявлениям и креативам
- Асинхронный консьюмер для RabbitMQ — сервис подписывается на события и записывает статистику в ClickHouse

**ClickHouse таблицы:** я создал три таблицы — `ads_stats`, `campaign_stats`, `creative_stats`. Они используют движок MergeTree, который позволяет эффективно хранить миллионы записей и быстро делать агрегацию.

## 🏎️ 🚀 🔥 Развертывание в production с Ansible

Я подготовил инфраструктуру для развёртывания на серверах с помощью Ansible. Это нужно для того, чтобы в будущем можно было автоматически настроить окружение: от установки Docker до развёртывания всех сервисов и мониторинга.

**Важный момент:** у меня нет удалённых серверов, и я не развёртывал это в production. Ansible — это задел на будущее, чтобы показать, как можно автоматизировать деплой.

Ansible использует шаблоны Jinja2 — это значит, что конфигурационные файлы генерируются автоматически на основе заданных переменных.

### 1. Установка зависимостей
```bash
cd infra/ansible
ansible-galaxy install -r requirements.yml
```

### 2. Настройка инвентаризации
Нужно отредактировать файл `inventory/production.yml` и указать IP-адреса ваших серверов.

### 3. Настройка переменных
В файле `group_vars/production.yml` я задал конфигурацию для production-окружения — тут можно указать размеры контейнеров, порты и другие параметры.

### 4. Запуск плейбука
```bash
ansible-playbook -i inventory/production.yml playbook.yml
```

### 5. Проверка развертывания
```bash
# Проверка запущенных контейнеров
ansible production -m shell -a "docker ps"

# Проверка работоспособности приложения
ansible production -m uri -a "url=http://localhost:8080/health"
```

### Что я запланировал для production

**Масштабируемость:** Ansible позволяет развернуть сервисы на нескольких серверах:
- Базы данных можно вынести на отдельные серверы
- Приложения запустить на разных хостах
- Мониторинг разместить на выделенном сервере

**Безопасность:**
- SSH-доступ к серверам
- Управление секретами через Ansible Vault
- Настройка брандмауэра
- Установка SSL/TLS сертификатов

**Производительность:**
- Оптимизация конфигурации под нагрузку
- Автоматическое масштабирование
- Выделение ресурсов под каждый сервис

**Мониторинг:**
- Prometheus собирает метрики со всех сервисов
- Grafana визуализирует данные
- AlertManager отправляет уведомления при проблемах

**Резервное копирование:**
- Базы данных автоматически бэкапятся
- Конфигурации сохраняются


## 📊 ClickHouse

ClickHouse я выбрал для хранения аналитики — он отлично справляется с большими объёмами данных и позволяет быстро агрегировать информацию.

**Какие таблицы я создал:**
- `ads_stats` — детальная статистика по каждому объявлению (показы, клики, стоимость)
- `campaign_stats` — агрегированные данные по кампаниям
- `creative_stats` — аналитика по креативам

Все таблицы используют движок MergeTree — он позволяет эффективно хранить данные и быстро выполнять запросы.

**Зачем это нужно:**
- Можно хранить миллионы записей о показах и кликах
- Быстро получать метрики: CTR, расходы, конверсии
- Анализировать эффективность рекламы по временным периодам

### Как я это использую:

**1. Создание базы данных и таблиц:**

Автоматически через скрипт:
```bash
bash infra/scripts/create_clickhouse_db.sh
```

Или вручную через curl:
```bash
# Создание базы данных analytics
curl -s -X POST 'http://localhost:8123/' -d "CREATE DATABASE IF NOT EXISTS analytics"

# Создание таблицы для статистики
curl -s -X POST 'http://localhost:8123/' -d "
CREATE TABLE IF NOT EXISTS analytics.ads_stats (
    date Date,
    campaign_id UInt64,
    ad_id UInt64,
    creative_id UInt64,
    impressions UInt64,
    clicks UInt64,
    cost Float64,
    timestamp DateTime
) ENGINE = MergeTree()
ORDER BY (date, campaign_id, ad_id)
"
```

**2. Загрузка и анализ данных:**
Таблицы оптимизированы для хранения миллионов записей. MergeTree движок обеспечивает быструю вставку и эффективные запросы с агрегацией.

**3. Интеграция с микросервисами:**
- `analytics-api` предоставляет REST API для доступа к данным
- Можно фильтровать по датам, кампаниям, объявлениям
- Данные загружаются асинхронно через RabbitMQ в реальном времени

## 🐰 RabbitMQ

RabbitMQ я использую для асинхронной обработки событий. Это позволяет сервисам не блокироваться друг на друге — например, когда создаётся объявление, событие уходит в очередь, а воркеры обрабатывают его в фоне.

**Как я настроил архитектуру:**
- **Обменник**: `ad_events` (тип topic) — маршрутизирует события
- **Очереди**: `ad_worker_queue` — воркеры забирают сообщения отсюда
- **Routing keys**: `ad.created`, `ad.updated` — разделяют типы событий

**Какие сервисы интегрированы:**

### Go-сервисы (`ads-api`)
- Файл `services/ads-api/internal/service/rabbitmq.go` — сервис для работы с RabbitMQ
- При создании или обновлении объявления автоматически публикуется событие
- Если публикация не удалась — операция не прерывается, ошибка просто логируется

### Python-сервисы
- `auth-api`: `services/auth-api/app/core/rabbitmq.py` — базовое подключение
- `analytics-api`: `services/analytics-api/app/core/rabbitmq.py` — асинхронный consumer для аналитики
- Воркеры: `services/workers/worker-py/worker.py` и `services/workers/worker-go/main.go` — обработка сообщений

### Как я это использую:

**1. Запуск RabbitMQ:**
```bash
# RabbitMQ доступен на порту 5672 (AMQP) и 15672 (Web UI)
# Web UI: http://localhost:15672 (логин: guest, пароль: guest)
```

**2. Публикация событий из ads-api:**
```go
// Автоматическая публикация при создании объявления
if s.rabbitmq != nil {
    if err := s.rabbitmq.PublishAdCreated(ctx, fmt.Sprintf("%d", ad.ID)); err != nil {
        // Логируем ошибку, но не прерываем операцию
    }
}
```

**3. Обработка сообщений в воркерах:**
```go
// Go воркер обрабатывает сообщения из очереди
func (w *AdEventWorker) processMessage(d amqp091.Delivery) {
    var data map[string]interface{}
    json.Unmarshal(d.Body, &data)

    eventType := data["type"].(string)
    switch eventType {
    case "ad_created":
        w.handleAdCreated(data)
    case "ad_updated":
        w.handleAdUpdated(data)
    }
}
```

**4. Воркеры:**
- Go worker (`services/workers/worker-go/main.go`) — обрабатывает события
- Python worker (`services/workers/worker-py/worker.py`) — тоже обрабатывает события
- Оба воркера подписаны на `ad_worker_queue` и обрабатывают `ad.created` и `ad.updated`

**5. Асинхронная аналитика:**
- `analytics-api` потребляет сообщения через асинхронный consumer
- Обработка через asyncio в фоновом режиме
- API не блокируется при обработке
- События `ad.created` и `ad.updated` записываются в ClickHouse для статистики

**6. Надёжность:**
- Все сервисы корректно завершают работу (graceful shutdown)
- Обработка ошибок подключения
- Автоматическое восстановление соединений
- Подтверждение обработки сообщений (ACK/NACK)

## 🔒 ACID-транзакции

Я реализовал полную поддержку ACID-транзакций для всех финансовых операций. Это было важно, потому что деньги не должны «теряться» или «появляться из ниоткуда».

### Атомарность (Atomicity)
Все операции выполняются в рамках одной транзакции. Если что-то идёт не так — происходит автоматический rollback. Гарантирую, что либо все изменения применяются, либо ни одно.

### Согласованность (Consistency)
Я использовал внешние ключи (FK) для поддержания целостности данных. Транзакции ссылаются на созданные объявления и креативы, бизнес-правила соблюдаются на уровне базы данных.

### Изолированность (Isolation)
Для предотвращения гонок я использую `SELECT ... FOR UPDATE`. Это блокирует строки на время финансовой операции — никто не может изменить баланс, пока транзакция не завершится.

### Долговечность (Durability)
Все изменения фиксируются в базе данных и сохраняются даже при сбоях системы. Финансовая история не потеряется.

### Как я это реализовал:
- **Python (auth-api)**: SQLAlchemy транзакции с `with_for_update()` — это основной сервис для работы с транзакциями
- **Связь сущностей**: `transaction_id` в таблицах `ads` и `creatives` — связывает транзакции с объявлениями и креативами
- **API endpoints**: `/billing/deposit`, `/billing/withdraw`, `/billing/charge-for-ad`, `/billing/charge-for-creative` — всё в auth-api
- **Тестирование**: транзакции протестированы на корректность и изолированность

## 🔑 Шифрование и безопасность

Я реализовал несколько уровней защиты данных в проекте.

### Хэширование паролей
- **Алгоритм**: bcrypt
- **Зачем**: защита от атак по словарю и rainbow-таблицам
- **Реализация**: функции `hash_password()` и `verify_password()` в `app/core/security.py`

### JWT-аутентификация
- **Алгоритм**: HS256 (HMAC с SHA-256)
- **Секретный ключ**: хранится в переменной окружения `SECRET_KEY`
- **Срок действия токена**: 60 минут (можно изменить через `ACCESS_TOKEN_EXPIRE_MINUTES`)
- **Реализация**: функция `create_access_token()` в `app/core/security.py`

### Защита данных
- Все чувствительные данные хранятся в `.env` файле
- `.env` добавлен в `.gitignore` — не попадёт в репозиторий
- Все входные данные проходят валидацию через Pydantic модели
